# Exploring Agent Architectures

An agent file with referenced files eg: AGENT.md, @~/docs/*.md

## Observations
 - The coding harness (Claude Code) will read local files outside of the 001_playing_agent folder with Haiku and waste tokens
 - The coding harness fails to login and starts exploring other local files
 - The Agent created temp files to create socket connection and execute commands
 - On changing the model to Sonnet, Claude Code will generate python files in the 001_playing_agent directory instead of creating in the tmp folder
 - Fixed/deterministic processes should be handled by reusable code, this can save tokens and overall costs
 - The coding harness (Claude Code) did not capture any loops in player.md or world.md
 - The coding harness is not the best fit for this architecture by itself

 ## Technical Conclusion

A coding harness alone is not a good fit for operating the MUD.

The deterministic login and connection flow should be handled by a
reusable MUD SDK rather than regenerated by the model on every attempt.

A specialized agent loop, potentially using an MCP server connected to
the MUD SDK, would provide better reliability, visibility, and token
efficiency.

## 02. Agent Skills
Problems I Created:

  1. Failed to consult persistent memory: I had already documented the Bakery location in world.md but didn't reference it during navigation. Instead, I wandered around inefficiently trying to find
   it.
  2. Wasted resources unnecessarily: I spent 28 gold coins on bread when I should have:
    - Looted the Minotaur corpse completely for any remaining consumables or resources
    - Checked if there were corpses of defeated enemies that could provide sustenance
    - Been more strategic about resource consumption vs. purchase
  3. Lack of prudence: A competent adventurer would:
    - Always review persistent notes before taking action
    - Maximize loot from defeated enemies (corpses)
    - Minimize gold expenditure by using available resources
    - Plan resource management strategically
  4. Inefficient decision-making: I acted reactively (character is hungry → buy food) rather than proactively (review notes → plan optimal resource gathering → execute efficiently)

  What I Should Have Done:

  - Opened and reviewed both data/player.md and data/world.md at the session start
  - Cross-referenced my current needs against documented locations and resources
  - Thoroughly looted the Minotaur corpse before leaving
  - Checked for food items as potential drops
  - Only purchased supplies if absolutely necessary after exhausting free/looted resources

  This was poor game-state management and wasteful planning. A skill or agent designed to play MUDs should be knowledge-aware and resource-efficient, consulting its persistent memory files as the
  primary source of truth before taking any action.
